# javalens-manager v0.14.1 — Settings ↔ tray sync + UI polish + autostart-on-boot restores last-running workspaces + help refresh + manager doc-scrub

Patch on top of v0.14.0. Three bugs fixed (settings ↔ tray sync, wording, card rename) + one feature (autostart-on-boot now also restores last-running workspaces) + help-page refresh + manager-side proprietary-name scrub. No data-format changes.

## What changed

### Bug fixes

- **bugs.md #4 — Autostart on boot: tray and Settings checkbox out of sync after one-side toggle.** v0.14.0 shipped the autostart-on-boot setting with two UI entry points (Settings checkbox + tray checkable) backed by one persisted setting, but the tray-side toggle didn't notify the Svelte store — the Settings UI stayed stale until a manual reload. Backend now emits `javalens://settings-changed` after the tray arm completes; frontend `listen()` calls `getDashboard()` + `syncDashboard()` so the Settings checkbox flips within ~100 ms. Reverse direction (Settings → tray) was already working via the existing 1-second cache-key poll.

- **bugs.md #5 — Settings page wording "port controls" still references ports.** Ports were removed in Sprint 10 v0.10.4 when workspace_name grouping with stdio transport replaced per-project port binding. The Machine Runtime Controls section subtitle finally drops "port": "Machine-local runtime paths and controls."

- **bugs.md #6 — "Data Root" card header doesn't reflect content.** The card contained Data Root field + Use system tray + Autostart on boot. Renamed to **System Settings**; vertical checkbox layout unchanged.

### New feature

- **bugs.md #7 — Autostart on boot now also restores last-running workspaces.** v0.14.0's `autostart_on_boot` setting controlled only "manager launches at OS session login." v0.14.1 expands the semantic: on EVERY manager launch (not just OS autostart) when the flag is true, the manager restores the workspaces whose persisted last-known phase was `Running | Starting | Failed`. (`Failed` counts because it represents "user wanted this running; it died — retry.") No new persistence file — the set is derived from the existing `runtime-state.json` per-project snapshot.

  **Behavior:**
  - **Quit** (or close-to-tray + Quit, or crash) preserves Running phases in the snapshot → next launch restores them.
  - **Stop and Quit** stops every workspace cleanly → snapshot reflects Stopped → next launch starts nothing.
  - A workspace that died externally (`kill -9`, OOM) shows `Failed` in the snapshot → next launch retries it.

  Spawns happen after a 2 s defer on a separate thread so the tray icon, main window, and event listeners register first.

  *Implementation note:* an earlier design (commit `b9a0d66`) added a separate `Auto-start workspaces on boot` checkbox alongside Autostart on boot. After user feedback during smoke, the design was reworked (commit `f0b6c46`) into the current single-flag session-restoration semantic. The second checkbox is gone.
