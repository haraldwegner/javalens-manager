# javalens-manager v0.17.0 — sharpened JavaLens-usage rule block

Minor release. Runtime unchanged; pairs with fork v1.9.0+.

## Why a minor bump

This release changes what the manager **deploys into your MCP clients** — the "rule block"
that the agent reads. The prior block was a single vague sentence ("prefer MCP calls before
filesystem grep") that agents routinely ignored, so you kept having to remind Claude/Cursor
to use JavaLens instead of `grep`/`sed`/`find` on Java code. This sharpens it into specific,
imperative guidance — a behaviour change in deployed clients, hence the minor bump.

## What changed

`build_rule_block` now emits two scannable sections (deployed into each client's rule file —
`CLAUDE.md`, `.cursor/rules/javalens-manager.mdc`, `AGENTS.md`, …):

1. **Java→JavaLens routing table** — an explicit "before `grep`/`rg`/`find`/`sed` or
   hand-reading `.java`, call the MCP tool" mapping:
   - symbol by name → `search_symbols`
   - who calls/uses → `find_references` / `find_method_references` / `get_call_hierarchy_incoming`
   - type shape / members / hierarchy → `analyze_type` / `get_type_members` / `get_type_hierarchy`
   - definition → `go_to_definition`
   - errors / compiles → `compile_workspace` + `get_diagnostics`
   - structural change → the refactoring tools (`rename_symbol`, `extract_method`, `pull_up`, …) — never hand-edit
   - Shell text-search is framed as a **fallback only** (JavaLens unavailable, or non-Java/non-semantic matches).
2. **Refactor in small, verified steps** — green baseline → ONE JavaLens refactoring
   (returns a diff + `undoChangeId`) → recompile + tests → keep, or `undo_refactoring`.

The block keeps its `<!-- javalens-manager:<client>:start/end -->` markers, so deploying
over an existing block replaces it in place (idempotent). **Existing users get the new rule
on their next "Deploy to Agents" click.**
