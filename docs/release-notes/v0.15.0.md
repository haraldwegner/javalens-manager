# javalens-manager v0.15.0 — bug #9 PRIMARY fix end-to-end (resident JVM hosting + URL-emitting MCP writer) + bug #8 stable jar path

Major release. Closes both open bugs on the tracker by switching the manager from "stdio MCP child per client" to **"one resident JVM per workspace + URL endpoints in deployed configs"**. Coupled with fork v1.8.5 (HTTP/SSE-default transport).

## What this fixes

### bug #9 — `autostartOnBoot: false` doesn't prevent javalens MCP servers from auto-starting (the 30 GB JVM leak)

**PRIMARY fix shipped here, end-to-end.** Pre-fix observed: 7 client sessions × 2 workspaces = **16 JVMs at ~24 GB RAM**.

**Architecture (paired with fork v1.8.5):**

- The manager now hosts ONE resident javalens JVM per workspace (Stage 10, `runtime_manager.rs`). The resident-spawn passes `-port <port> -token <token>` so the fork v1.8.5 HTTP listener binds where the manager expects.
- Deployed MCP-client configs (`~/.cursor/mcp.json`, `~/.claude.json`, etc.) now write **URL endpoints** instead of stdio commands (Stage 11, `manager_service.rs`):

  ```json
  {
    "mcpServers": {
      "jl-jats-orb-ws": {
        "url": "http://127.0.0.1:8800",
        "headers": { "Authorization": "Bearer <64-hex-token>" }
      }
    }
  }
  ```

- Per-workspace `(port, token)` pairs are allocated once (range 8800–8999) and persisted in `projects.json` so they survive manager restarts (Stage 9, new `resident` module).
- N MCP clients × M workspaces → **M JVMs total**, not N × M. With 3 Claudes + Cursor open: 2 JVMs instead of 16.

**Autostart-honoring (new `WriterMode` enum):**

- `autostart_on_boot = true` → URL entries written with `disabled: false`. Clients connect.
- `autostart_on_boot = false` + `WriterMode::Remove` (default) → managed entries stripped from deployed configs entirely. New clients see no MCP server.
- `autostart_on_boot = false` + `WriterMode::Disable` → entries written with `disabled: true`. Visible-but-inert; user toggles on with one click.

The `autostart` toggle now does what it says, end-to-end.

### bug #8 — Auto-downloaded fork jar breaks deployed MCP configs because deployed `args` reference a versioned jar path

Replaced with a stable `~/.cache/javalens-manager/tools/javalens/current/` symlink (POSIX atomic rename-into-place; Stage 8, `release_manager.rs`). `runtime.json` records `jar_path` as `current/<jar-relative>` instead of the versioned form. The single-cached-runtime invariant is preserved — old `javalens-*` dirs get cleaned up after the symlink swap commits.

Windows symlinks need Developer Mode; on platforms where `update_current_symlink` returns `Err`, the install falls back to the versioned form (degrades the bug-#8 mitigation but doesn't fail the install). Full Windows path is folded into the Sprint 16 Windows installer track.

## Strategic positioning

- The new "manager hosts the JVM" model is what makes the multi-client benefit observable: **one JVM serving Claude + Cursor + (future) Antigravity in parallel**, not N private children. Documented as a USP vs upstream Pzalutski-pixel/javalens-mcp in the fork v1.8.5 release notes.
- The same architecture sidesteps Antigravity's well-known per-conversation MCP-child orphaning bug ([discuss.ai.google.dev #139866](https://discuss.ai.google.dev/t/bug-all-mcp-server-processes-are-orphaned-after-conversation-ends-accumulate-indefinitely-causing-system-memory-exhaustion/139866)): the resident JVM is owned by the manager, not the IDE, so conversation lifecycle no longer leaks JVMs.

## Other changes

- Default release-repo URL bumped from `hw1964/javalens-mcp` to `haraldwegner/javalens-mcp` (the post-rename URL). `read_settings` rewrites the legacy `hw1964/...` value transparently on startup so existing v0.14.x users pick up the v1.8.5 release without manually editing `settings.json`.

## Pairs with

- **Fork v1.8.5** ([haraldwegner/javalens-mcp release notes](https://github.com/haraldwegner/javalens-mcp/releases/tag/v1.8.5)) — HTTP/SSE as the default transport, READY-line contract, Bearer auth. v0.15.0 will refuse to start the resident if the cached fork jar is older than v1.8.5 (no `-port`/`-token` support pre-1.8.5).

## Migration notes

- **Settings.json**: `release_repo` auto-rewrites from `hw1964/javalens-mcp` to `haraldwegner/javalens-mcp` on first read. No manual edit needed.
- **Projects.json**: gains a new `workspaces` array tracking per-workspace `(resident_port, resident_token)`. Older files load cleanly via `#[serde(default)]` and get the new field populated lazily on first resident-spawn.
- **Deployed MCP configs**: the next "Deploy to agents" run rewrites every managed entry from stdio command/args to URL endpoint. Existing v0.14.x stdio entries in `~/.cursor/mcp.json` etc. will become broken pointers (the fork v1.8.5 default is HTTP, not stdio); deploy once after upgrading to migrate them.
- **Open Cursor / Claude windows**: restart them after the deploy so they re-read the new URL entries.
