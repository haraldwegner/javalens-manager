# javalens-manager v0.16.2 — Linux scroll fix (the real one)

Single-issue patch. Runtime unchanged; pairs with fork v1.9.0.

## Smooth scrolling on Linux (#20)

v0.16.1 tried to fix the Linux scroll jank with CSS (dropping sticky `backdrop-filter`
blurs, adding `contain: paint`). **That wasn't the cause** — scrolling stayed jumpy. This
release fixes it for real.

**What it actually was:** on Linux the webview is WebKitGTK, and on hybrid-GPU stacks
(here: Intel Alder Lake iGPU + NVIDIA dGPU, X11, WebKitGTK 2.50.4) the WRY/Tauri webview's
accelerated-compositing path initialises only partially — used, but broken enough to make
scrolling jump. The tell: a native GTK WebKitGTK app (GNOME's `yelp`) scrolls perfectly
smoothly on the *same* WebKitGTK, which isolated the problem to the webview runtime, not
our page. (See [tauri#10566](https://github.com/tauri-apps/tauri/issues/10566).)

Systematically ruled out first — none were the cause: GPU/compositor env vars (DMABUF
on/off, Intel-vs-NVIDIA PRIME offload), box-shadow rasterisation, the body gradient + panel
translucency, and inner-scroller-vs-document-scroll layout.

**The fix:** set `WEBKIT_DISABLE_COMPOSITING_MODE=1` before the webview is created
(`lib.rs::run()`, `cfg(target_os = "linux")`, overridable via the same env var). Turning
WebKitGTK accelerated compositing off entirely drops to a clean, consistently-smooth scroll
path. It's set in the app entry point, so it covers `tauri dev` and every Linux package
(AppImage / .deb) uniformly — no reliance on a per-format CI bake. macOS and Windows are
untouched (different webviews; both were already smooth).

The ineffective v0.16.1 CSS tweaks are left in place (harmless, near-invisible) rather than
churned out.

## Verification

`cargo build` clean; confirmed smooth live in dev on the affected hybrid-GPU box.
