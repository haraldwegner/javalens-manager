# javalens-manager v0.15.1 — Sprint 15 post-ship hotfix bundle

Patch on top of v0.15.0. Bundles five discovered-after-ship issues that surfaced once real Claude Code / Cursor / Antigravity sessions tried to consume the v0.15.0 + fork v1.8.5 release pair end-to-end. v0.15.0 is functional but trips one of these on a fresh install; v0.15.1 is the version to install.

Pairs with fork v1.8.6 (MCP `protocolVersion` negotiation hotfix, same release window). Install both together.

## What was broken

### 1. Service probe didn't pass `-transport stdio` → "received non-JSON output from JavaLens stdout: READY url=…"

The Settings → "Test services" probe spawns a short-lived JVM and pipes JSON-RPC over stdio. v0.15.0 dropped the `-transport stdio` flag in the spawn args, so against fork v1.8.5 (HTTP/SSE default) the probe JVM bound an HTTP port and printed `READY url=… token=…` on stdout — the probe parser couldn't read JSON-RPC out of that and rejected the probe as "non-JSON output".

End-user symptom: Settings showed every workspace's service probe red with the `received non-JSON output from JavaLens stdout: READY url=…` error, even though the resident JVMs were perfectly healthy and the dashboard listed them as Running.

### 2. Deploy returned "No deployable services could be resolved" when `autostart_on_boot` was off

Stage 11's "autostart=off + Remove mode → strip entries" branch in `build_deploy_servers` was misdirected. With v0.15.0's URL semantics the deploy entry just points at `http://127.0.0.1:<port>`; whether a resident JVM is currently listening there is the resident-service lifecycle's concern, not the writer's. The old "stdio-args auto-spawn on client connect" hazard (the original bug #9 framing) is gone — URL clients get connection-refused if the resident isn't up; they don't spawn anything themselves. The auto-strip just stopped the user from deploying anything while autostart was off.

End-user symptom: every Deploy click with autostart off → "No deployable services could be resolved" toast and no file written.

### 3. Post-write validator rejected URL entries as "missing non-empty command"

`validate_client_config_shape` was written when entries had stdio `command` + `args`. v0.15.0 flipped the writer to emit `url` + `headers.Authorization`, but the validator still required `command` to be a non-empty string and `args` to be a non-empty string array — both absent from URL entries. The deploy write succeeded; the post-write self-check then deleted what it had just written and reported "server '<id>' missing non-empty command".

End-user symptom: every Deploy click with autostart on → toast "server 'jl-orb-strategies' missing non-empty command" (or similar per server) and the rewritten config got rolled back.

### 4. Writer URL omitted the `/mcp` path → Claude Code `/mcp` showed connectors as `failed`

The fork's HTTP transport serves the MCP endpoint at `/mcp` (POST) and `/mcp/events` (SSE GET). The writer was emitting the bare base URL `http://127.0.0.1:<port>` without any path. Claude Code hits the URL as given, gets `404 Not Found` from the root, and marks the connector as `failed` immediately. Quick verification:

```
$ curl -sS -o /dev/null -w "%{http_code}\n" http://127.0.0.1:8805/
404
$ curl -sS -o /dev/null -w "%{http_code}\n" http://127.0.0.1:8805/mcp
405                ← 405 because GET; POST is the real path
```

End-user symptom: every javalens connector in Claude Code's `/mcp` view as `failed`, even with `type: "http"` and a valid `Authorization: Bearer …` header. Resident JVMs themselves were healthy and the dashboard listed them as Running.

### 5. Writer omitted `type: "http"` → Claude Code `/mcp` showed connectors as `failed` with "command: expected string, received undefined"

Claude Code's MCP-config parser switches on the `type` field. Without `type`, it falls through to its stdio schema, which mandatorily reads `command` — and emits "command: expected string, received undefined" against a URL entry. Per the Claude Code MCP docs, the spec form is `{ "type": "http", "url": "...", "headers": {...} }`.

End-user symptom: `/mcp` in Claude Code listed every javalens connector as `failed` immediately on session start, with the above error in the diagnostics panel. Cursor was more tolerant and worked without `type`.

## What's fixed

- **Probe spawn** (`manager_service.rs`): pinned `-transport stdio` in the JDK probe command. Probes now run against fork v1.8.5/v1.8.6 in stdio mode regardless of the default transport.
- **`build_deploy_servers`** (`manager_service.rs`): decoupled from `autostart_on_boot`. Deploy now writes entries unconditionally. `WriterMode::Disable` still writes `disabled: true` when autostart is off; `WriterMode::Remove` no longer strips on user-initiated deploy (use the explicit "Delete" deploy action instead).
- **Post-write validator** (`manager_service.rs`): `validate_client_config_shape` rewritten to validate URL entries — checks `url` is a non-empty string and `headers.Authorization` is a `Bearer <token>` value. Stdio shape no longer recognised because the writer no longer emits it.
- **MCP-config writer** (`manager_service.rs`, both writer sites — `write_managed_json_block` and `build_client_mcp_json`): emits `type: "http"` as the first field of every entry, alongside `url` + `headers.Authorization`. This is the schema Claude Code expects; Cursor accepts it too.
- **URL construction** (`manager_service.rs::build_deploy_servers`): writer now emits `http://127.0.0.1:<port>/mcp` (with the `/mcp` path) instead of the bare base URL. Matches the fork's HTTP transport endpoint at `/mcp` POST.

## What didn't change

- The resident-JVM hosting model is unchanged from v0.15.0 (one JVM per workspace; lifecycle owned by `runtime_manager.rs`).
- Stable `current/javalens.jar` symlink (bug #8 fix, v0.15.0) unchanged.
- Port + token allocation, `projects.json` schema, release-poller flow — all unchanged.
- No new CLI flags, no new settings, no new tray items.

## Migration

Drop-in compatible with v0.15.0. On first launch, v0.15.1 rewrites every deployed client config to add `type: "http"`. Claude Code and Cursor windows already open need to reconnect (close + reopen, or restart the IDE) to pick up the new entry shape.

If you were running the fork at v1.8.5 against manager v0.15.0 and saw the `received non-JSON output from JavaLens stdout: READY url=…` probe failure, that's resolved by v0.15.1 even without bumping the fork. But install fork v1.8.6 alongside — it fixes the MCP `protocolVersion` negotiation issue that made Claude Code drop the connection at `initialize` time.
