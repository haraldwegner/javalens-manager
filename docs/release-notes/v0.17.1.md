# javalens-manager v0.17.1 — activate Lombok comprehension

Patch release. Pairs with **fork v1.10.0**, which bundles `lombok.jar` in the product; this
release is the manager half that actually turns Lombok comprehension on.

## What changed

The manager now **detects Lombok in a workspace** (a build file referencing
`org.projectlombok` / `lombok:lombok`, or a `lombok.config`) and, only for those workspaces,
prepends `-javaagent:<product>/lombok.jar` to the resident JavaLens JVM.

That agent patches the Eclipse compiler so JavaLens's analysis tools *see* Lombok-synthesized
members — `analyze_type` / `get_type_members` list `@Data`/`@Getter` accessors,
`find_references` finds them, and `compile_workspace` stops reporting phantom errors on
Lombok-using code.

- **Conditional** — non-Lombok workspaces launch exactly as before (no agent, no overhead).
- **Graceful** — if the runtime predates fork v1.10.0 (no bundled `lombok.jar`), the agent is
  silently skipped rather than failing the launch.

Proven end-to-end against the built product: with the agent, `get_type_members` on a `@Data`
class surfaces the generated getters; without it (control), they're absent.

## Verification

97 `cargo test --lib` green, including 7 new tests for Lombok detection (pom/Gradle/
`lombok.config`/submodule), the bundled-jar lookup, and the "needs both a Lombok project AND
a bundled jar" gate on the `-javaagent` argument.
